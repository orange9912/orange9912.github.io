V8 这个词，很多前端人应该都耳熟能详，我也看过不少相关的文章，正好想写一篇文总结一下在 V8 这个引擎下，JS 是怎么运行的。

# V8 那点事

它的历史来源，我就不写了，网上也有很多资料。

V8 的源码：https://source.chromium.org/chromium/chromium/src/+/master:v8/

但主要在了解 V8 引擎之前，需要先有一点编译原理的知识。

V8 使用 C++ 开发，并在谷歌浏览器中使用。在运行 JavaScript 之前，相比其它的 JavaScript 的引擎转换成字节码或解释执行，V8 将其编译成原生机器码（IA-32, x86-64, ARM, or MIPS CPUs），并且使用了如内联缓存（inline caching）等方法来提高性能。有了这些功能，JavaScript 程序在 V8 引擎下的运行速度媲美二进制程序。

V8 可以独立运行，也可以 嵌入 到任何 C++ 应用程序。项目托管在 Google Code 上 [1] ，基于 BSD 协议，任何组织或个人可以将其源码用于自己的项目中。———— 百度百科

# V8 如何运行 JavaScript

在编译原理里，一般分为五个阶段：

  1. 词法分析，生成 token 流
  2. 语法分析，生成 AST（语法抽象树）
  3. 语义分析
  4. 优化
  5. 生成代码

在 V8 中，差不多是这样：

  1. 从网络上下载 js 脚本，解析得到 js 源代码文本
  2. 经过 Parser，词法分析和语法分析，得到 AST
  3. 将 AST 送入 Ignition，生成字节码，然后边解释边执行。
  4. 高频执行的会被标记为热点，通过编译器编译成机器码，下次直接执行机器码。

大体的过程是这样，但是中间又有很多小细节。

# Parsing（解析）

Parser，会将 js 源代码解析成 token、再解析成 AST，这个就不再叙述了，可以自行看编译原理。

主要是在语法分析中分为两个阶段：

  • Pre-Parser。

    预解析，他会跳过一些未被使用的代码，比如函数的声明。在这个过程，他仅仅解析语法、声明的作用域等，速度是 Full-parser 的 2 倍

    并且不会产生 AST

  • Full-Parser

    解析使用的代码，产生 AST。

# Interpret（解释）

在解释阶段，Compiler 会接收之前生成的 AST，并转换成字节码，然后一行一行将字节码解释成机器码运行。

运行的过程中，会将重复执行、需要优化的代码进行热点标记，通过更高级的编译器优化后再编译。

其实在 V8 设计的初期，是直接将 AST 编译成机器码、中间没有字节码这么个东西的。但是因为直接编译成机器码来执行的话，大小往往是源代码的几十倍，占用了不小的内存空间。在移动设备（内存较小）流行时，这样的缺点非常突出。

于是后来就加上了字节码。和机器码相比，字节码不仅占用的空间小了许多，还让他有了更强的跨平台能力，因为机器码是和平台有着强依赖关系的，但字节码没有。

# 编译器

上面提到通过编译器编译成机器码并运行，主要是指 V8 的 TurboFan 编译器。

在 V8 中，字节码配合解释器 + 编译器，又被称为 JIT（即时编译技术），类似于 Java 虚拟机的技术。

解释器在解释执行字节码时，会收集代码信息，标记一些热点代码，** 热点代码 (hot spot)** 就是一段代码被重复执行多次，TurboFan 会将热点代码直接编译成机器码，保存起来，下次调用直接运行对应的二进制的机器码，加速执行速度。

# 存在的问题...

或者说其实是 Javascript 这门语言存在的问题，不是 V8 的问题

V8 的 JIT 在优化时，只能针对静态类型的变量。比如说执行一个函数，函数需要的两个参数类型是确定的，这种情况就可以优化。

但是问题就在于，js 是一门动态的语言，类型在执行的时候可能会变化，会导致 JIT 失效。就需要重新进行 Parser->Compiler->Interpreter->Execution，前两步正是最耗费时间的步骤，这也是为什么 js 没法搞一些高性能应用，比如大型游戏。

所以为了保持最大限度的 JIT 优化，大佬们就搞了一个语言 ——WebAssembly（简称 WASM），这是一门可以直接和机器码打交道的汇编语言，当然，这是后面的了，有兴趣可以看看：https://mp.weixin.qq.com/s/dEOIArtK6DIfewIva2zLKw

# V8 如何优化对象的存储

JavaScript 中，对象是一个非常重要的概念。在 V8 之中，做了大量的优化。

一个对象通常拥有许多熟悉，例如 { a: 1, b: 2 } ，你可能会发现，用 for-in 遍历对象时，输出的顺序总是一致，并不是随机

在 V8 之中，对象主要分为两种属性：Named Properties（命名属性）和 Indexed Properties（数组索引属性，又称为 elements）

这两种数据结构一般来说是线性的，但有时候也可能会被切换到字典形式的存储。

# 数组索引属性（Elements）

  • 定义时键是整型的属性
  • 数值升序排列
  • 存储在对象的 elements 属性中

很明显，elements 主要用于各种 Array.prototype 方法，如 pop、slice。

空数组预分配的大小为 4。

# 带孔的数组

带孔的数组就是 [1,,3] 这种，充满的数组就是 [1,2,3] 。

我们都知道，当查找一个对象中的元素，如果在该对象中元素不存在，会顺着原型链查询。在 V8 内通过对数组是否全部充满来判断，如果数组是带孔的，查不到对应的值，就会顺着数组的原型链查找。

如果已经是充满的，就不会往原型链上查询，而是直接在当前作用域里查询。

# 快数组

就是线性数据结构存储的数组。

# 慢数组

使用字典存储的数组

# 快数组 VS 慢数组？

当出现这种情况，如果用快数组，就会相当的浪费空间

  • Fast 模式的存储结构是 FixedArray 并且长度小于等于 elements.length，可以通过 push 和 pop 扩容和缩容数组（弱语言类型，无法固定空间，只能通过 length 变化实现），实现内存空间的连续
  • slow 模式的存储结构是一个以数字为键的 HashTable（哈希表），不用开辟连续的内存空间，节省内存
  • 快数组就是以空间换时间的方式，申请了大块连续内存，提高了执行效率。
  • 慢数组以时间换空间，不必申请连续的空间，节省了内存，但需要付出效率变差的代价。

# 快慢转换

  • 快 -> 慢

    1. 如果快数组新的容量 >= 3 * 扩容后的容量 * 2，意味着它比 HashTable 形式存储占用更大的内存，快数组会转换为慢数组；
    2. 如果快数组新增的索引与原来最大索引的差值大于等于 1024，中间全部是_hole 标记，快数组会被转换会慢数组。

  • 慢 -> 快

    元素能存放在快数组中并且长度不在 smi 之间（64 位，-231 到 232-1），并且当前慢数组空间相比快数组节省值小于等于 50%，则转变成为快数组。

# 命名属性（Named Properties）

  • 定义时键是非整型的属性
  • 定义时的顺序排列
  • 存储在对象的 properties 属性中

在命名属性中不是简单的使用键来推断在 Properties 中的位置，还需要存储额外的数据，这就是隐藏类。

# 隐藏类

在 V8 中，每个对象都有关联的一个 HiddenClass（隐藏类），因为对象属性的类型和值是可以随时改变的，在运行时才能拿到属性内存偏移量，为了提高性能，才有了隐藏类。

每次对象的属性发生改变时，会随着对象的变化而动态更新内存偏移量等信息。

隐藏类中记录了对象中每个属性的标识信息 (descriptors)。它保存了属性 key 以及描述符数组的指针。描述符数组包含了有关命名属性的信息，例如名称本身以及值保存的位置，但只会存命名属性相关的。

  • 具有相同结构（相同属性定义顺序相同）的对象共享相同的 Hidden Class
  • 默认情况下，添加的每个新命名属性都会导致创建一个新的 Hidden Class。
  • 添加数组索引属性不会创建新的 Hidden Class。

# 对象内属性

V8 支持所谓的对象内属性，这些属性直接存储在对象本身上。

当对象的属性数量小于 10 个的情况，直接将属性 key 存在对象内的属性上，如果需要查询某个 key 的值时，直接中对象内中获取对应 key 的值就可以了。

这也是 V8 中最快的属性。

# 快属性

fast properties。一般来说我们把存储在线性数据结构中的属性定义为快速。

# 慢属性

slow propeties。不存储在线性数据结构，而是存储在属性字典中。

# 快属性 VS 慢属性？

  • 快属性的信息仍然存储在 Hidden Class，慢属性存在字典中，不再通过 Hidden Class 共享。
  • 慢属性新增删除较快。快属性如果大量的新增和删除，会耗费大量时间在维护 Hidden Class 中。

# 命名属性总结

  • 有三种不同的命名属性类型：in-object、fast 和 slow/dictionary。
    1. 对象内属性直接存储在对象本身上并提供最快的访问。
    2. 快速属性存在于属性存储中，所有元信息都存储在 HiddenClass 上的描述符数组中。
    3. 慢属性存在于一个自包含的属性字典中，元信息不再通过 HiddenClass 共享。
  • 慢速属性允许有效地删除和添加属性，但访问速度比其他两种类型要慢。
  • 快速属性存在 inline cache（内联缓存）优化，所以访问速度相对快。

# V8 如何实现数组的排序

Array.prototype.sort 方法。

在 ECMAScript 不同版本规范对 Array.prototype.sort 的定义中没有要求用什么样的排序方式实现 sort() 方法，也没有要求是否要采用稳定算法，都是靠各大浏览器自己实现。

V8 中使用的是快速排序和插入排序的结合。对于长度 <= 10 的数组使用的是插入排序 (稳定排序算法) ， >10 的数组使用的是快速排序。快速排序是不稳定的排序算法。

顺带一提，别的引擎，如 Nitro 使用的是归并排序和桶排序。再顺带一提，C++ 中的 sort 使用的是快速排序 + 堆排序 + 插入排序。

# V8 如何进行垃圾回收

垃圾回收是个很经典的话题，各个语言都有自己的垃圾回收机制。

主要是分为手动和自动，像 C++ 就是手动，js 就是自动垃圾回收。

但是注意：自动垃圾回收并不是代表我们开发者不需要关注内存管理！

Orinoco，V8 的垃圾回收器名字。

不过在进行介绍 V8 垃圾回收方式之前，先说说 js 的内存机制吧。

# 内存空间

js 中的内存空间主要分为以下三种：

  • 代码空间，用来存储可执行代码的地方。
  • 栈空间。栈空间，调用栈中用来存储调用上下文的地方，而且基本类型也存储在这里，引用类型存放一个指针在这里，指向堆空间。
  • 堆空间。用来存放引用类型。

# 栈空间回收

javascript 在执行一个函数的时候，会把它的上下文环境压入执行栈中，并且 ESP（Extended Stack Pointer，扩展栈指针）上移指向当前栈顶。

函数执行完成后，直接将 ESP 下移，下次压栈会直接覆盖 ESP 以上的位置，就等于是回收了。

# 题外话：尾调用优化

当函数执行完成时，才能够销毁，但我们经常一个函数调用另外一个函数，这样执行栈就会不断压入，很多时候，执行尾调用优化可以减少栈空间的使用，复用一个上下文。

# 闭包的内存模型

  • 当执行 outer 时，会创建一个上下文。
  • 在编译时，遇到内部函数，进行词法分析，同时发现内部函数引用外部变量，就会判断到是一个闭包。
  • 然后会在堆空间创建一个对象，用于存放引用到的外部变量。
  • 然后即使 outer 执行完毕，内部函数也仍然引用着创建的堆空间对象。

# 堆空间回收

堆空间回收就相对来说复杂很多，但其实每个语言都大同小异，都是根据已经提出的一些论文来改进的

# 代际假说

代际假说主要有两个：

  • 大部分对象在内存中的存在时间很短。
  • 那些一直存活的对象通常会存在很久

简单来说，就是对象的生存对象通常两极分化。javascript 把内存分成了两块区域：新生代和老生代。

# 新生代垃圾回收

在 V8 中，由副垃圾回收器来负责新生区的垃圾回收。

为什么是副呢？因为新生区比较小，通常在 1～8M 左右，而老生区相对来说就大的多了。

V8 之中，对内存的使用有限制（64 位下约 1.4G、32 位下约 0.7G）。

当然，也可以在启动时传递 --max-old-space-size 或 --max-new-space-size 来调整这个限制大小。但是只能在初始化 / 启动时指定，无法运行过程中扩充。

新生代的垃圾回收算法主要通过 Scavenge 算法进行垃圾回收，该算法的流程主要是： 将堆内存一分为二，一个处于使用中 (from)，一个处于闲置中 (To)。

实际上也分为 nursery 和 intermediate ，经历过一次 GC 的就叫 intermediate 子代。

分配内存时，会先分配到使用中的内存（from 空间），当开始进行垃圾回收的时候，回收生命周期结束的对象。

  1. 对对象区域进行垃圾回收（标记清除），留下在使用中的活跃对象
  2. 把对象区域中剩下的活跃对象复制到空闲区域（复制过程会消除内存碎片）
  3. 翻转，使用区域（from）和空闲区域 (To) 身份互换。循环重复

这个算法是典型的牺牲空间换时间的算法（因为只能使用堆内存的一半）。

该算法只复制存活的对象，而实际应用中新生代大多数对象的存活时间都是比较短的，所以时间效率上相对来说比较优异。但是每次都需要复制，而复制操作需要时间成本，如果新生区空间设置得太大了，那么每次清理的时间就会过久，所以为了执行效率，一般新生区的空间会被设置得比较小（1～8m）。

# 对象晋升策略

当一个对象经过新生代垃圾回收中多次复制仍然存活时，它就会被认为是生命周期较长的对象，随后会被移动到老生代内存空间中（也就是晋升）。

在上面新生代垃圾回收过程中，并不是每一次都会把对象复制到闲置空间（To）中，而是在复制之前，会先判断两个条件，如果满足条件，则会触发对象晋升，而不是复制到闲置空间中。

  • 是否经历过 Scavenge 回收（主要通过地址判断）
  • To 空间的内存占用率是否超过一定的限制

达成其一，则实行对象晋升，晋升到老生代，由主垃圾回收器处理。

# 老生代垃圾回收

在老生代之中，大部分都是存活时间比较久的对象。这时候咱们在进行大量的复制，时间效率会低很多。因此不适合使用 Scavenge 算法。

而 V8 之中，使用的是 **Mark - Sweep （标记 - 清除）和 Mark - Compact （标记 - 整理）** 相结合的方式。

# Mark - Sweep（标记 - 清除）

标记 - 清除分为两个阶段：

  • 标记阶段。遍历堆中的所有对象，并标记活着的对象。
  • 清除阶段。清除没有被标记的对象。

标记 - 清除标记活着的对象，并且只清理死亡的对象。Scavenge 算法只复制活着的对象，前者适合存活多、后者适合存活少的地方。

但是清理得多了，会存在很多内存碎片，碎片多到一定程度，当空间不足以分配空间给新生代晋升的对象时，V8 就会采取 Mark - Compact 来消除内存碎片。

# Mark - Compact（标记 - 整理）

Mark - Compact 是在 Mark - Sweep 的基础上演变过来的。它和 Mark - Sweep 的区别主要在于，当对象在标记为死亡后，在整理的过程中，将活着的对象往一端移动（整理），移动完成后，直接清理边界外的内存。

但是因为需要先移动后清除，很显然，它比 Mark - Sweep 要慢。

# 总结

回收算法速度碎片适用场景
Scavenge快无（复制过程中消除）新生代 (存活对象少的地方)
Mark-Sweep较慢有老生代 (存活对象多的地方)
Mark-Compact最慢无老生代

V8 根据代际假说，将对象分为两部分：新生代与老生代。并且对两部分使用不同的算法

  • 新生代使用 Scavenge 算法进行垃圾回收，这种算法牺牲空间来换取时间，实际上只利用了新生代一半的空间进行存储。但优点在于面对生命周期短的对象时速度较快，而且不会产生内存碎片。
  • 新生代的对象满足两个条件之一，就会进行对象晋升，移动到老生代区域
  • 老生代使用 Mark-Sweep 及 Mark-Compact 两种结合。主要使用 Mark-Sweep，当碎片太多、不足以分配一块内存给晋升的对象时，就利用 Mark-Compact 进行整理。

# 优化

为了避免出现回收逻辑和 JavaScript 逻辑不一致的情况，当垃圾回收进行时，需要暂停 JavaScript 的应用逻辑，这种现象叫做全停顿。

在老生代的垃圾回收中，因为空间比较大，运行一次垃圾回收可能要占用相当长的一段时间，这段停顿时间可能会造成卡顿。

而在 Orinoco 中（V8 的垃圾回收器），它利用并行、增量和并发技术来进行垃圾收集、释放主线程。

为了减小停顿时间带来的影响，V8 引入了增量标记，即将一个同步一口气完成的动作分为许多个 “step”，在这些 step 之间可以穿插一些 JavaScript 应用逻辑（其实我总觉得和操作系统调度、React fiber 的改进思想有些异曲同工，都是分片），这样就不会长时间无响应。
